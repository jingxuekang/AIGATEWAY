# AI Gateway 架构与运转流程

## 项目定位

**AI Model Gateway** 是一个 AI API 中转网关，作为用户和多个 AI 服务商之间的中间层。

### 核心价值

1. **统一接口**：用户只需对接一个 API，即可使用多个 AI 服务商的模型
2. **统一计费**：统一的配额管理和计费系统
3. **负载均衡**：自动在多个渠道间分配请求
4. **成本优化**：根据价格和性能选择最优渠道
5. **监控审计**：完整的调用日志和使用统计

## 详细运转流程

### 阶段 1：管理员配置（一次性）

```
管理员
  │
  ├─► 1. 在 DeepSeek 官网申请 API Key (sk-35f472760e4343e0adf054f839db7e70)
  │
  ├─► 2. 在 Gateway Admin 后台配置渠道
  │      - 渠道名称: DeepSeek-Primary
  │      - Provider: deepseek
  │      - API Key: sk-35f472760e4343e0adf054f839db7e70
  │      - Base URL: https://api.deepseek.com
  │
  ├─► 3. 配置模型定价
  │      - 模型: deepseek-chat
  │      - 输入价格: ¥1/1M tokens
  │      - 输出价格: ¥2/1M tokens
  │
  └─► 4. 创建用户账号
         - 用户名: user001
         - 初始配额: 1,000,000 tokens
```

### 阶段 2：用户申请 API Key

```
普通用户
  │
  ├─► 1. 登录 Gateway 前端 (http://localhost:3001)
  │
  ├─► 2. 提交 API Key 申请
  │      - Key 名称: "生产环境 Key"
  │      - 申请模型: deepseek-chat, gpt-4
  │      - 申请理由: "用于聊天机器人项目"
  │
  ├─► 3. 等待管理员审批
  │
  └─► 4. 获得 Gateway API Key
         ✅ gw-user-abc123xyz (这是用户的 Key，不是 DeepSeek 的)
```

### 阶段 3：用户调用 API

```
用户应用
  │
  ├─► 1. 发送请求到 Gateway
  │      POST http://localhost:9080/v1/chat/completions
  │      Headers:
  │        Authorization: Bearer gw-user-abc123xyz
  │      Body:
  │        {
  │          "model": "deepseek-chat",
  │          "messages": [{"role": "user", "content": "你好"}]
  │        }
  │
  ▼
Gateway Core 处理
  │
  ├─► 2. 验证用户 Key (gw-user-abc123xyz)
  │      - 检查 Key 是否存在
  │      - 检查 Key 是否启用
  │      - 检查用户配额是否充足
  │      - 检查是否有 deepseek-chat 权限
  │
  ├─► 3. 选择上游渠道
  │      - 找到支持 deepseek-chat 的 Provider
  │      - 选择 DeepSeek-Primary 渠道
  │      - 获取渠道的 API Key (sk-35f472760e4343e0adf054f839db7e70)
  │
  ├─► 4. 转发请求到 DeepSeek
  │      POST https://api.deepseek.com/v1/chat/completions
  │      Headers:
  │        Authorization: Bearer sk-35f472760e4343e0adf054f839db7e70
  │      Body:
  │        {
  │          "model": "deepseek-chat",
  │          "messages": [{"role": "user", "content": "你好"}]
  │        }
  │
  ▼
DeepSeek API 响应
  │
  └─► 5. 返回响应
         {
           "id": "chatcmpl-xxx",
           "model": "deepseek-chat",
           "choices": [{
             "message": {
               "role": "assistant",
               "content": "你好！我是 DeepSeek..."
             }
           }],
           "usage": {
             "prompt_tokens": 5,
             "completion_tokens": 20,
             "total_tokens": 25
           }
         }
  │
  ▼
Gateway Core 后处理
  │
  ├─► 6. 计算费用
  │      - 输入: 5 tokens × ¥1/1M = ¥0.000005
  │      - 输出: 20 tokens × ¥2/1M = ¥0.00004
  │      - 总计: ¥0.000045
  │
  ├─► 7. 扣减用户配额
  │      - 用户配额: 1,000,000 → 999,975 tokens
  │
  ├─► 8. 记录调用日志
  │      - 用户: user001
  │      - 模型: deepseek-chat
  │      - Token 消耗: 25
  │      - 费用: ¥0.000045
  │      - 延迟: 1.2s
  │      - 状态: success
  │
  └─► 9. 返回响应给用户
         (与 DeepSeek 的响应相同)
```

## 关键概念对比

### 上游 API Key vs Gateway API Key

| 类型 | 持有者 | 用途 | 示例 |
|------|--------|------|------|
| **上游 API Key** | 管理员 | Gateway 调用上游服务商 | `sk-35f472760e4343e0adf054f839db7e70` |
| **Gateway API Key** | 普通用户 | 用户调用 Gateway | `gw-user-abc123xyz` |

### 配额管理

```
管理员在上游服务商的配额:
  DeepSeek 账户余额: ¥100
  ↓
  可用 Token: 约 50,000,000 tokens

Gateway 内部配额分配:
  用户 A: 1,000,000 tokens
  用户 B: 2,000,000 tokens
  用户 C: 500,000 tokens
  ↓
  总分配: 3,500,000 tokens (< 50,000,000)
```

## 多渠道负载均衡示例

```
用户请求: deepseek-chat
  │
  ▼
Gateway 查找可用渠道:
  ├─► DeepSeek-Primary (权重: 70, 状态: 正常)
  ├─► DeepSeek-Backup  (权重: 30, 状态: 正常)
  └─► DeepSeek-Fallback (权重: 10, 状态: 禁用)
  │
  ▼
根据权重选择:
  70% 概率 → DeepSeek-Primary
  30% 概率 → DeepSeek-Backup
  │
  ▼
如果 Primary 失败:
  自动切换到 Backup
```

## 计费流程

### 1. 预付费模式（当前实现）

```
用户充值 ¥100
  ↓
获得 50,000,000 tokens 配额
  ↓
每次调用扣减配额
  ↓
配额用完后无法调用
  ↓
需要再次充值
```

### 2. 后付费模式（可扩展）

```
用户调用 API
  ↓
记录使用量
  ↓
月底生成账单
  ↓
用户支付账单
```

## 优势总结

### 对用户的优势

1. ✅ **一个 Key 访问多个模型**：不需要去每个服务商申请
2. ✅ **统一计费**：一个账户管理所有 AI 服务
3. ✅ **简化开发**：统一的 API 格式
4. ✅ **成本透明**：清晰的价格和使用记录
5. ✅ **高可用**：自动故障转移

### 对管理员的优势

1. ✅ **集中管理**：统一管理所有用户和渠道
2. ✅ **灵活定价**：可以加价或打折
3. ✅ **成本控制**：精确的配额管理
4. ✅ **数据分析**：完整的使用统计
5. ✅ **商业化**：可以作为 SaaS 服务运营
